# ビジネスユースケース - React Job Board Application

## BUC001: 🏢求人情報検索🌊

**使用時のコンテキスト**: 求職者が新しい職業機会を探索する際、または採用担当者が市場の求人動向を調査する際に実行される。

**スコープ**: 求人掲載プラットフォーム（企業）

**レベル**: ユーザー目的

**主アクター**: 求職者

**利害関係者と利益**:
- **求職者**: 希望条件に合致する求人を効率的に発見できる
- **一般ユーザー**: 市場動向や業界情報を収集できる
- **採用担当者**: 自社求人の競合状況を把握できる
- **プラットフォーム運営者**: ユーザー エンゲージメントの向上

**事前条件**:
- 求人情報データベースが存在し、アクセス可能である
- 検索システムが稼働している
- ユーザーがプラットフォームにアクセス可能である

**最低保証**: 
- 検索結果が適切に表示される（該当なしの場合も含む）
- システムが応答不能になることはない

**成功保証時**: 
- 求職者が希望条件に合致する求人情報を取得し、次のアクションに進むことができる

**トリガー**: 求職者が求人検索画面にアクセスし、検索を実行する

**主成功シナリオ**:
1. 求職者が求人検索画面にアクセスする
2. システムが検索インターフェースを表示する
3. 求職者が検索条件（職種、勤務地、給与等）を入力する
4. 求職者が検索ボタンをクリックする
5. システムが検索条件を検証する
6. システムがデータベースから該当する求人情報を取得する
7. システムが検索結果を一覧形式で表示する
8. 求職者が検索結果を確認し、興味のある求人を選択する

**拡張**:
- 3a 高度な検索条件を使用: 求職者が詳細フィルター（経験年数、企業規模等）を設定する
- 4a 検索条件が空の場合: システムが全件検索を実行する
- 6a 該当する求人が存在しない場合: システムが「該当なし」メッセージを表示し、関連する求人を提案する
- 7a 検索結果が多数の場合: システムがページネーションまたは無限スクロールを提供する

**技術およびデータのバリエーションリスト**:
- 3a デバイス別インターフェース: モバイル、タブレット、デスクトップで最適化された入力方法
- 7a 表示形式のバリエーション: リスト表示、カード表示、地図表示

**関連情報**:
- React Sample 解析結果での features/jobs/api/get-jobs.ts の実装パターンを参考
- 検索パフォーマンス要件: 3秒以内の応答時間
- モバイルファースト設計による UX 最適化

## BUC002: 🏢求人詳細確認🌊

**使用時のコンテキスト**: 求職者が興味を持った求人について詳細情報を確認し、応募判断を行う際に実行される。

**スコープ**: 求人掲載プラットフォーム（企業）

**レベル**: ユーザー目的

**主アクター**: 求職者

**利害関係者と利益**:
- **求職者**: 求人の詳細な情報を把握し、適切な応募判断ができる
- **採用企業**: 求人内容を正確に伝達し、適格な応募者を獲得できる
- **プラットフォーム運営者**: ユーザーの滞在時間とエンゲージメント向上

**事前条件**:
- 求人情報が公開状態である
- 求職者が検索結果または直接リンクから求人を特定している

**最低保証**: 
- 求人の基本情報（職種、企業名、勤務地）が表示される
- システムが適切にエラーハンドリングを行う

**成功保証時**: 
- 求職者が求人の全詳細情報を確認し、応募するかどうかの判断を下すことができる

**トリガー**: 求職者が求人一覧から特定の求人を選択する

**主成功シナリオ**:
1. 求職者が求人一覧で興味のある求人をクリックする
2. システムが求人詳細ページにナビゲートする
3. システムが求人の詳細情報を取得する
4. システムが求人詳細情報を構造化して表示する
   - 企業情報（名前、業界、規模）
   - 職務内容（責任、要求スキル、経験）
   - 雇用条件（給与、勤務時間、福利厚生）
   - 応募方法と締切
5. 求職者が情報を確認し、応募ボタンまたは戻るボタンを選択する

**拡張**:
- 3a 求人が存在しない場合: システムが404エラーページを表示し、関連求人を提案する
- 3b 求人が非公開の場合: システムがアクセス制限メッセージを表示する
- 4a 企業詳細情報の表示: 求職者が企業名をクリックして企業プロフィールページに移動する
- 5a お気に入り登録: 求職者が後で参照するために求人をブックマークする

**技術およびデータのバリエーションリスト**:
- 4a レスポンシブ表示: デバイスサイズに応じた情報の配置とレイアウト調整
- 4b 多言語対応: ブラウザ言語設定に基づく表示言語の切り替え

**関連情報**:
- React Sample の features/jobs/components/public-job-info 実装を参考
- SEO 要件: 求人詳細ページの検索エンジン最適化
- アクセシビリティ: WCAG 2.1 AA準拠

## BUC003: 🏢求人応募🌊

**使用時のコンテキスト**: 求職者が応募したい求人を決定し、正式な応募手続きを行う際に実行される。

**スコープ**: 求人掲載プラットフォーム（企業）

**レベル**: ユーザー目的

**主アクター**: 求職者

**利害関係者と利益**:
- **求職者**: 簡潔で確実な応募プロセスを通じて希望職種に応募できる
- **採用企業**: 応募者の情報を体系的に収集し、選考プロセスを開始できる
- **採用担当者**: 応募者情報の一元管理と効率的な選考進行
- **プラットフォーム運営者**: 応募完了率の向上とプラットフォーム価値の増大

**事前条件**:
- 求人が応募受付中の状態である
- 求職者がプラットフォームにログインしている（任意）
- 応募フォームが適切に機能している

**最低保証**: 
- 応募情報が確実に送信される
- 応募者と採用企業に確認通知が送られる

**成功保証時**: 
- 求職者の応募が正常に受理され、採用企業の選考プロセスに登録される
- 応募者が応募完了確認と今後のプロセスについて通知を受ける

**トリガー**: 求職者が求人詳細ページで応募ボタンをクリックする

**主成功シナリオ**:
1. 求職者が求人詳細ページで「応募する」ボタンをクリックする
2. システムが応募フォームを表示する
3. 求職者が必要情報を入力する
   - 基本情報（氏名、連絡先）
   - 履歴書ファイルのアップロード
   - 志望動機の記入
4. 求職者が入力内容を確認する
5. 求職者が応募を送信する
6. システムが入力内容をバリデーションする
7. システムが応募情報をデータベースに保存する
8. システムが求職者に応募完了メールを送信する
9. システムが採用担当者に新規応募通知を送信する
10. システムが応募完了画面を表示する

**拡張**:
- 2a 未ログインユーザーの場合: システムがゲスト応募フォームまたはログイン案内を表示する
- 6a バリデーションエラーの場合: システムがエラー内容を明示し、修正を促す
- 6b ファイルサイズ制限超過: システムがファイルサイズエラーを表示し、圧縮または分割を案内する
- 7a データベースエラーの場合: システムが一時的な保存を行い、後で再試行する
- 8a メール送信失敗の場合: システムが代替通知手段（プラットフォーム内通知）を使用する

**技術およびデータのバリエーションリスト**:
- 3a ファイル形式バリエーション: PDF、Word、テキスト形式の履歴書対応
- 3b 入力方法バリエーション: フォーム入力、LinkedIn 連携、外部プロフィール取り込み
- 8a 通知方法バリエーション: メール、SMS、プッシュ通知

**関連情報**:
- React Sample の features/jobs/api/create-job.ts のパターンを応募API に応用
- プライバシー要件: GDPR、個人情報保護法への準拠
- セキュリティ要件: ファイルアップロードのウイルススキャン

## BUC004: 🏢求人作成🌊

**使用時のコンテキスト**: 採用担当者が新しいポジションの募集を開始する際、または既存求人の改訂版を作成する際に実行される。

**スコープ**: 採用管理プラットフォーム（企業）

**レベル**: ユーザー目的

**主アクター**: 採用担当者

**利害関係者と利益**:
- **採用担当者**: 効率的で正確な求人作成により採用活動を迅速に開始できる
- **組織管理者**: 組織の求人活動を監督し品質を保持できる
- **求職者**: 詳細で魅力的な求人情報により適切な応募判断ができる
- **人事マネージャー**: 統一されたフォーマットで求人品質を管理できる

**事前条件**:
- 採用担当者が適切な権限でログインしている
- 組織情報が登録され、有効な状態である
- 求人作成に必要なテンプレートまたは項目設定が準備されている

**最低保証**: 
- 求人の基本情報が保存される
- システムが適切なエラーハンドリングを提供する

**成功保証時**: 
- 完全な求人情報が作成され、公開準備が整った状態になる
- 求人が検索可能な状態でデータベースに保存される

**トリガー**: 採用担当者がダッシュボードで「新規求人作成」ボタンをクリックする

**主成功シナリオ**:
1. 採用担当者がダッシュボードで「新規求人作成」をクリックする
2. システムが権限を確認し、求人作成フォームを表示する
3. 採用担当者が求人基本情報を入力する
   - 職種名、部門、勤務地
   - 雇用形態、給与範囲
   - 必要スキル、経験年数
4. 採用担当者が職務詳細を記述する
   - 業務内容、責任範囲
   - 求める人物像
   - 福利厚生、キャリアパス
5. 採用担当者が応募条件を設定する
   - 応募締切、選考プロセス
   - 連絡方法、必要書類
6. 採用担当者が入力内容をプレビューで確認する
7. 採用担当者が保存または公開ボタンをクリックする
8. システムが入力内容をバリデーションする
9. システムが求人情報をデータベースに保存する
10. システムが作成完了を通知し、次のアクション（公開、編集）を提案する

**拡張**:
- 3a テンプレート使用: 採用担当者が既存求人や企業テンプレートから情報をコピーする
- 6a 組織管理者承認が必要: システムが承認待ち状態で保存し、管理者に通知する
- 8a バリデーションエラー: システムが不足項目や不正入力を指摘し、修正を促す
- 9a 自動下書き保存: システムが定期的に入力内容を一時保存する

**技術およびデータのバリエーションリスト**:
- 3a 入力支援機能: 自動補完、候補提示、リアルタイムバリデーション
- 4a エディター機能: リッチテキストエディター、マークダウン対応
- 6a プレビュー表示: 求職者視点での表示確認

**関連情報**:
- React Sample の features/jobs/components/create-job-form の実装を参考
- テンプレート管理: 業界別、職種別の標準テンプレート提供
- 承認ワークフロー: 組織ポリシーに応じた段階的承認プロセス

## BUC005: 🏢求人編集🌊

**使用時のコンテキスト**: 採用担当者が既存求人の情報更新、条件変更、または募集停止を行う際に実行される。

**スコープ**: 採用管理プラットフォーム（企業）

**レベル**: ユーザー目的

**主アクター**: 採用担当者

**利害関係者と利益**:
- **採用担当者**: 市場状況に応じた柔軟な求人条件調整が可能
- **組織管理者**: 求人活動の適切な管理と品質維持
- **求職者**: 最新で正確な求人情報へのアクセス
- **人事マネージャー**: 統一された求人管理プロセス

**事前条件**:
- 編集対象の求人が存在し、アクセス権限がある
- 求人が編集可能な状態である（削除されていない）
- 採用担当者が適切な権限でログインしている

**最低保証**: 
- 既存の求人情報が保持される
- 編集履歴が記録される

**成功保証時**: 
- 求人情報が正確に更新され、変更内容が即座に反映される
- 必要に応じて関係者に変更通知が送信される

**トリガー**: 採用担当者が求人一覧から特定の求人の編集ボタンをクリックする

**主成功シナリオ**:
1. 採用担当者が求人一覧で編集したい求人を選択する
2. 採用担当者が「編集」ボタンをクリックする
3. システムが編集権限を確認する
4. システムが現在の求人情報を取得し、編集フォームに表示する
5. 採用担当者が必要な項目を修正する
6. 採用担当者が変更内容をプレビューで確認する
7. 採用担当者が「保存」または「公開」ボタンをクリックする
8. システムが変更内容をバリデーションする
9. システムが更新された求人情報をデータベースに保存する
10. システムが変更完了を通知し、更新内容を確認表示する

**拡張**:
- 3a 編集権限がない場合: システムが権限エラーを表示し、管理者への連絡方法を案内する
- 4a 求人が削除済みの場合: システムがエラーメッセージを表示し、求人一覧に戻る
- 5a 重要項目の変更: システムが承認要求フローを開始する
- 9a 並行編集の競合: システムが他の編集者による変更を検出し、競合解決を促す

**技術およびデータのバリエーションリスト**:
- 5a 変更差分表示: 変更前後の内容をハイライト表示
- 9a バージョン管理: 編集履歴の保存と過去バージョンへの復元機能

**関連情報**:
- 編集権限管理: ロールベースアクセス制御
- 変更通知: 応募済み求職者への変更内容通知
- 監査ログ: 編集操作の詳細記録

## BUC006: 🏢応募者管理🌊

**使用時のコンテキスト**: 採用担当者が受信した応募を管理し、選考プロセスを進行する際に実行される。

**スコープ**: 採用管理プラットフォーム（企業）

**レベル**: ユーザー目的

**主アクター**: 採用担当者

**利害関係者と利益**:
- **採用担当者**: 効率的な応募者スクリーニングと選考進行
- **組織管理者**: 採用活動の進捗と品質の監督
- **人事マネージャー**: 統一された選考プロセスの管理
- **応募者**: 透明で迅速な選考プロセス

**事前条件**:
- 求人に対する応募が存在する
- 採用担当者が該当求人への管理権限を持っている
- 応募者情報が適切に保存されている

**最低保証**: 
- 応募者の基本情報が表示される
- 選考状況が適切に記録される

**成功保証時**: 
- 応募者の選考状況が体系的に管理され、効率的な採用判断が実行される
- 応募者に適切なフィードバックが提供される

**トリガー**: 採用担当者がダッシュボードで応募者管理画面にアクセスする

**主成功シナリオ**:
1. 採用担当者が応募者管理画面にアクセスする
2. システムが権限を確認し、管理可能な求人の応募者一覧を表示する
3. 採用担当者が特定の求人または応募者を選択する
4. システムが応募者の詳細情報を表示する
   - 基本情報、履歴書、志望動機
   - 応募日時、選考状況
5. 採用担当者が選考アクションを選択する
   - 書類通過、面接設定、不採用
   - メモ追加、評価入力
6. 採用担当者が選考結果を入力し、確定する
7. システムが選考状況を更新する
8. システムが応募者に結果通知を送信する（設定による）
9. システムが更新された応募者一覧を表示する

**拡張**:
- 3a フィルタリング: 採用担当者が選考状況、応募日などで応募者を絞り込む
- 5a 一括操作: 採用担当者が複数応募者に対して同一アクションを実行する
- 6a 承認プロセス: 重要な選考判断に組織管理者の承認が必要
- 8a 通知カスタマイズ: 選考段階に応じた個別メッセージの作成

**技術およびデータのバリエーションリスト**:
- 4a 情報表示形式: リスト表示、カード表示、カンバン形式
- 5a 評価システム: 星評価、数値評価、カテゴリ別評価

**関連情報**:
- プライバシー保護: 応募者情報の適切な管理と保護
- 選考プロセス標準化: 公平で一貫した評価基準
- 通知システム: 自動通知とカスタムメッセージの管理
